## Problem-Solving Frameworks

**5 Whys**
```
Problem: [symptom]
Why 1: [cause] → Why 2: [deeper cause] → ... → Root cause
```

**Fishbone (Ishikawa)**
```
Categories: People, Process, Technology, Environment, Materials, Methods
For each: identify contributing factors
```

**TRIZ Inventive Principles**
Key principles for technical contradictions:
- Segmentation, Extraction, Local quality
- Inversion, Universality, Nesting
- Dynamics, Partial/Excessive action
- Dimension change, Mechanical vibration

**Theory of Constraints**
```
1. Identify the constraint (bottleneck)
2. Exploit the constraint (maximize its output)
3. Subordinate everything else
4. Elevate the constraint
5. Repeat
```

## Problem Analysis Process

```
1. Problem Statement — what is observed, not interpreted
2. Context Mapping — stakeholders, systems, constraints
3. Root Cause Hunt — 5 Whys / Fishbone
4. Contradiction Analysis — what two things conflict?
5. Solution Space — generate options (TRIZ / reframe)
6. Trade-off Analysis — evaluate against constraints
7. Recommendation — with rationale
```

## Key Questions

- What do you observe? (not what you think is happening)
- What changed right before this started?
- Who is affected and who is not?
- What would the system look like without this problem?
- What constraint, if removed, would make this trivial?

## Rules

- Symptoms vs root causes — never treat symptoms as root causes
- Deductive first: observe → hypothesize → test → conclude
- Right question > fast answer — slow down to speed up
- Challenge the problem statement itself (maybe solving wrong problem)
- No emojis (Mantra IA-23)
- Source claims with evidence (Fact-Check protocol)
